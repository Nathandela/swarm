$ swarm

Every coding agent on your machine, in one keyboard-driven view — running in the background, surviving the terminal. This page is the navigable design: screens, flows, architecture, lifecycle, and the test strategy.

Design intent

NEEDS INPUT

The group that matters. Always first, always highlighted. A session lands here when its agent asks a question or requests permission.

WORKING

Agents mid-task. The one-line summary shows the last meaningful output so you know what each is doing without attaching.

READY FOR REVIEW

The agent finished its turn. Attach, review the diff, send the next prompt — the session drops back to Working.

COMPLETED

Exited sessions, exit code shown. They stay listed (transcript on disk) until you delete them with x.

Attach is raw passthrough: the agent CLI's own interface, untouched, full-screen. swarm adds exactly one thin line — session name and the detach key — and even that is toggleable. No fluff.

The TUI is a thin client, and the daemon itself is replaceable: each session lives under its own tiny shim process that owns the PTY. Close the terminal, kill the daemon, brew upgrade swarm — the agents keep running and the next daemon reconnects to the shims (ADR-001). This beats Agent View, where a daemon restart ends your sessions.

One binary, two roles: swarm is the TUI; it auto-spawns swarm daemon on first run. The protocol between them is the product's spine — the V2 mobile client will speak the same one.

PROTOCOL IRREVERSIBLE

Control plane: versioned NDJSON. Data plane: length-prefixed binary frames for PTY bytes (JSON can't carry them). Exclusive attach lease. Changes require an ADR (002).

DISK SCHEMA IRREVERSIBLE

meta.json per session is the truth (atomic writes, schema_version); roster is a rebuildable index. Survives crashes by construction (ADR-003).

VT EMULATOR HARDEST PIECE

Each shim maintains the session's real screen grid. Attach paints a snapshot of it — raw byte replay can't reconstruct a full-screen TUI. Status heuristics read the grid too (ADR-002).

ADAPTERS HIGH CHURN

Typed events first: Claude hooks, Codex app-server, Gemini hooks, OpenCode events. Grid parsing is the last resort. Every adapter is preceded by a characterization harness recording the real CLI.

Session lifecycle

Status is three independent dimensions — process (running / exited / lost), turn (active / idle / unknown), interaction (none / prompt / permission) — and the four view groups are derived from them. Detection prefers typed signals (Claude hooks, Codex app-server events); grid heuristics are the fallback; a staleness guard turns silence into unknown, never into a confident lie.

Test strategy

Manifesto discipline: tests are the specification, written first. Two keystones: a fake agent (scripted binary that prints, asks, idles, exits — every daemon/shim flow testable deterministically, no API keys) and the characterization harness — before any adapter is built, it records the real CLI's behavior (PTY output per state, hook/event payloads, version) into a fixture corpus that becomes the adapter's acceptance baseline.

CHARACTERIZATION

Per CLI, before its adapter exists: record real fixtures for working / asking / idle / exit, capture hook and app-server event payloads, produce the capability-matrix entry. Re-run on every CLI upgrade.

UNIT

Adapter signal rules against recorded fixtures; VT emulator golden-grid tests (feed bytes, assert final grid) + fuzz on split escape sequences; status derivation; protocol frame round-trips; meta.json schema round-trips.

COMPONENT

TUI via teatest golden files; daemon handlers over an in-memory socket pair; shim against the fake agent.

INTEGRATION

Real daemon + shims + fake agent over real UDS: launch → events → attach (snapshot + live) → resize → kill (process group). Daemon kill -9 → restart → shims survive, reconnect (scenarios 10–11). Disk-full, slow-subscriber, lease-conflict injections.

Traceability

Every EARS requirement maps to at least one test; the scenario table in the spec is the acceptance suite skeleton. Invariants (e.g. “a client disconnect never kills a session”, “roster on disk is always parseable by the previous minor version”) get their own property tests. Fixture corpus grows whenever an agent CLI ships a UI change that breaks a parser — the fix commits the new recording (never answer the same question twice).

The adapter fixture corpus is the project's most valuable test asset: real recorded terminal output of each agent CLI in each state (working, asking, idle). It is what makes “agent-agnostic” a tested claim instead of a hope.

Requirements (EARS)

The system spec in docs/specifications/system-spec.md is the source of truth. Summary by area:

D — Daemon & shim lifecycle 8 reqs
  • D-1 Any swarm command auto-starts the daemon if absent (setsid, stdio to log).
  • D-2 Each session's PTY is owned by a dedicated shim process, not the daemon.
  • D-3 Closing the launching terminal never affects sessions.
  • D-4 On start: rebuild registry from per-session metadata, reconnect live shims, mark gone ones lost (PID + start-time verified).
  • D-5 Daemon crash or upgrade: sessions continue under shims, next daemon reconnects, nothing lost.
  • D-6 0700 state dir, 0600 socket; flock before bind; stale sockets unlinked under the lock.
  • D-7 Second daemon exits on lock failure; its client retries the winner.
  • D-8 Version-skew handshake failure → clear error + safe `swarm daemon restart`.
S — Sessions 7 reqs
  • S-1 Launch: shim execs adapter-composed argv (never a shell string) in its own process group.
  • S-2 Persisted: id, agent, dir, options, captured env, timestamps, status, shim PID+start-time, native conversation id, exit code, schema_version.
  • S-3 Worktree isolation (later epic): branch swarm/<id>, teardown on delete.
  • S-4 Kill signals the whole process group (MCP servers don't leak).
  • S-5 Shim maintains the VT grid + capped, rotated transcript; spinner frames collapsed.
  • S-6 Launch carries the client's environment — agents never inherit the daemon's stale env.
  • S-7 Configurable max concurrent sessions with clear rejection.
P — Protocol 6 reqs
  • P-1 Control plane: versioned NDJSON. Data plane: length-prefixed binary frames. One UDS, defined framing + max frame size.
  • P-2 Control ops: handshake, list, launch, kill, delete, attach/detach, resize, subscribe. Data: PTY I/O, grid snapshots.
  • P-3 Status events pushed to subscribers; slow subscribers disconnected, never blocking PTY drain.
  • P-4 Client disconnect mid-attach never harms the session; lease released.
  • P-5 Exclusive controller lease with generation ids; stale input/resize rejected.
  • P-6 Daemon re-validates all client-supplied input server-side.
V — General view 6 reqs
  • V-1 Groups: Needs input / Working / Ready for review / Completed.
  • V-2 Status events reflected ≤ 1 s, no user action.
  • V-3 Keyboard-only: ↑↓, ⏎ attach, esc back, ctrl+x kill (confirm), n new.
  • V-4 Row = agent, dir, status, elapsed, one-line summary.
  • V-5 needs_input / review transitions surface an in-TUI notification.
  • V-6 Claude Code minimalism; no mouse required.
L — Launch flow 3 reqs
  • L-1 Form: dir (free text, ~ expansion), agent, model/options, optional prompt, worktree toggle.
  • L-2 Only PATH-detected CLIs offered; missing ones greyed with install hint.
  • L-3 Invalid dir = inline error, no launch.
A — Attach 5 reqs
  • A-1 Raw mode (IXON off) full-screen passthrough; every key forwarded except detach.
  • A-2 Ctrl+\ detaches (configurable; Ctrl+Q rejected — XON collision); session continues.
  • A-3 Controlling client's resize propagates to the session PTY (lease authority).
  • A-4 Attach paints a serialized VT-grid snapshot, then the live stream — never raw byte replay, never blank.
  • A-5 Chrome ≤ one thin line, toggleable off.
T — Adapters & status 7 reqs
  • T-1 One adapter interface: detect + version range, launch args, declarative options schema, signal sources, resume + conversation-id extraction.
  • T-2 Typed signals first: Claude hooks, Codex app-server, Gemini hooks, OpenCode events — authenticated with per-session tokens, installed per-invocation.
  • T-3 Grid heuristics (emulated screen, never raw bytes) as fallback.
  • T-4 Staleness guard: silence while "active" degrades to unknown — never confidently wrong.
  • T-5 New adapter = zero changes to core/protocol/TUI.
  • T-6 Characterization harness records the real CLI before its adapter is built; fixtures are the acceptance baseline.
  • T-7 v1.0: Claude Code + Codex. v1.1: Gemini, OpenCode, AGY (after characterization).
R — Persistence 3 reqs
  • R-1 XDG state dir (0700): meta.json per session = truth (atomic, schema_version); roster = rebuildable index; transcripts 0600, capped.
  • R-2 Resume creates a new session linked via resumed_from, using the captured native conversation id.
  • R-3 Completed sessions listed until deleted (delete also tears down worktrees).
N — Non-functional 7 reqs
  • N-1 TUI first paint < 100 ms p95 (daemon warm, ≤ 50 sessions).
  • N-2 Attach adds < 10 ms local latency p95.
  • N-3 Event-driven; PTYs always drained; spinner churn collapsed; near-zero client-idle CPU.
  • N-4 Single static binary (CGO_ENABLED=0, client+daemon+shim roles); Homebrew + go install; macOS/Linux; no tmux.
  • N-5 Repo follows the agentic-codebase-manifesto (vendored).
  • N-6 Faithful ANSI/alt-screen passthrough; hostile escapes (OSC 52) filtered.
  • N-7 Documented limitation: machine sleep pauses agents (keep-awake is v1.x).
F — V2-forward 2 reqs
  • F-1 Protocol versioned + capability-negotiated; endpoint ids + namespaced session ids from release one.
  • F-2 Remote transport is NOT claimed v1-ready — it gets its own auth/threat-model ADR; v1 only guarantees no UDS-specific schema assumptions.
